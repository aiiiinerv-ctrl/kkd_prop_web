# Storage keys namespaced `public/…` vs `private/slips/…`, never Next's `public/` dir

Uploaded files (portfolio images, payment slips) are served exclusively through `src/app/files/[...key]/route.ts` against a driver-abstracted `STORAGE_ROOT`, keyed by a `public/…` or `private/slips/…` prefix — never written into Next's own `public/` directory. This is a boundary decision, not an implementation detail: payment slips are financial documents tied to a real customer and must never become fetchable by a guessable static URL, which is exactly what Next's `public/` dir would allow. Splitting by key prefix (rather than, say, a separate S3 bucket per sensitivity level) keeps the driver interface simple while still making the access-control boundary explicit and greppable — any code path that writes a file must decide its namespace up front, and any reviewer can `grep` for `private/` to audit exposure.
