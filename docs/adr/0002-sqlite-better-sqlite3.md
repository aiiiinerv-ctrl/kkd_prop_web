# SQLite (via better-sqlite3 adapter) as the database

The site runs on Prisma 7 + SQLite through the `better-sqlite3` driver adapter, not Postgres. This is a deliberate choice for a small-to-medium marketing site with a modest, predictable write volume (leads, admin content edits) — SQLite means zero external database service to provision, back up, or pay for, and the whole app can run as a single container with the DB file on disk. The trade-off, accepted knowingly: no horizontal scaling of writes and no built-in replication, which would matter if lead volume or concurrent admin usage grew far beyond what's expected. `DATABASE_URL` in `.env.example` is deliberately commented "swap to a postgres:// URL later if migrating" — Prisma's adapter model keeps that door open, but it's not a trivial swap (SQLite-specific type/migration quirks would need auditing). Don't "fix" this to Postgres without confirming the scale assumption has actually changed.
